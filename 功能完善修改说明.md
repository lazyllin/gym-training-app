# 健身记录 App 功能完善修改说明

生成日期：2026-06-27  
参考范围：当前 `gym_record_app` Flutter 项目、`26.06.26APP开发.md` 笔记、同类训练记录 App 公开资料。

## 1. 修改目标

当前 App 已经具备本地离线记录训练、动作库、训练详情、基础统计、JSON 导出的雏形。下一阶段目标不是把它做成社交型或 AI 教练型产品，而是先把“真实训练时好用、事后能复盘、数据可迁移”做好。

核心目标：

- 训练中录入更顺手：每组是否完成、做完时间、动作耗时、复制上一组、休息提示都要清楚。
- 数据更准确：动作部位、器械、训练容量、时长、RPE 等字段有明确含义。
- 复盘更有用：日历总览、动作趋势、部位训练频率、重量容量和自重/时间类训练量能看懂。
- 本地数据更可控：能看到保存位置，能导入旧数据，能导出备份。

不优先做：

- 登录、社交动态、排行榜、云同步、广告、会员体系。
- 自动生成训练计划。可以作为远期功能，但不影响第一轮功能完善。

## 2. 当前项目现状

主要结构：

- `lib/main.dart`：App 入口和底部导航。
- `lib/app_state.dart`：全局状态，管理训练记录和动作库。
- `lib/services/storage_service.dart`：本地 JSON 读写、导出和清空。
- `lib/models/`：训练记录、动作记录、组记录、动作定义。
- `lib/pages/add_workout_page.dart`：新增/编辑训练主流程。
- `lib/widgets/set_input_row.dart`：每组训练数据输入。
- `lib/widgets/exercise_picker_sheet.dart`：动作选择和自定义动作。
- `lib/services/stats_service.dart`：统计计算。

当前已支持：

- 新增/编辑/删除训练记录。
- 选择日期、训练主题、部位、状态、时长、备注。
- 添加动作，按动作类型记录重量、自重、时间、有氧、灵活性。
- 自定义动作和动作库开关。
- 首页、历史、统计、设置四个页面。
- JSON 备份导出和清空训练记录。

当前主要问题：

- 每组的完成开关没有文字说明，用户会疑惑“蓝色滑块是什么”。
- RPE 字段和次数同时出现，但没有解释，容易被误认为是重复概念。
- 每组没有完成时间，训练中容易忘记上一组是否做过。
- 动作只有大类 `category`，缺少更准确的主练/辅助部位。
- 动作缺少器械信息，无法区分杠铃、哑铃、固定器械、龙门架等。
- 训练总时长主要依赖手填，没有从每组/每动作自动推算。
- 没有日历视图。
- 没有旧数据导入。
- 设置页没有展示本地文件保存位置。

## 4. 需求判断

### 4.1 RPE 不建议直接删除

笔记问题：新增训练的动作里，每一组都涉及到次数和 RPE，但是这两个是一个概念吗？

结论：不是同一个概念。

- 次数：这一组实际完成了多少次。
- RPE：这一组主观强度有多高，通常用 1-10 表示。力量训练里常和 RIR 关联，例如 RPE 8 大致表示还剩 2 次力竭前余量。

修改建议：

- 保留 RPE，但改为“高级可选字段”。
- UI 文案从 `RPE` 改为 `强度 RPE`，提示语写 `可选，1-10；10 表示接近力竭`。
- 默认可以折叠或由设置开关控制，避免每次记录都打扰。
- RPE 只对 `weighted` 和 `bodyweight` 默认显示；`cardio/timed/mobility` 暂不默认显示。

验收标准：

- 用户能看懂 RPE 不是次数。
- 不填 RPE 也能保存。
- 旧数据中的 `rpe` 字段继续兼容。

### 4.2 “蓝色滑块”要改成明确的完成状态

当前 `SetInputRow` 里的 `Switch` 实际对应 `SetRecord.completed`。

修改建议：

- 将裸 `Switch` 改为带文字的 `SwitchListTile`、`Checkbox` 或横向状态按钮。
- 文案：`已完成` / `未完成`。
- 完成状态切换时联动记录完成时间。

验收标准：

- 页面上不能只出现一个无解释的开关。
- 关闭后该组仍保留数据，但统计默认不计入未完成组。

### 4.3 每组记录时间

笔记需求：每一组添加时记录时间，避免忘记做没做过。

修改建议：

给 `SetRecord` 增加字段：

- `createdAt`：组被添加的时间。
- `completedAt`：组被标记完成的时间。
- `updatedAt`：最后修改时间。

行为规则：

- 新增一组时写入 `createdAt`。
- 用户点“已完成”时，如果 `completedAt` 为空，则写入当前时间。
- 用户改回“未完成”时，保留 `createdAt`，清空或隐藏 `completedAt`，统计不计入。
- 详情页显示每组完成时间，例如 `20:31 完成`。

验收标准：

- 训练中能看到每组是否完成以及完成时间。
- 未完成组不参与容量、次数、时长统计。
- 老数据导入后没有时间也能正常展示。

### 4.4 每个动作记录时间，自动给出总时间

笔记需求：每一个动作记录时间，最后给出总时间。

修改建议：

给 `ExerciseRecord` 增加字段：

- `startedAt`
- `finishedAt`
- `restSec`

计算规则：

- 动作开始时间：该动作第一组 `createdAt` 或 `completedAt`。
- 动作结束时间：该动作最后一组 `completedAt`。
- 训练自动时长：所有动作最早开始到最晚结束的差值。
- 仍保留手动 `durationMin`，作为用户可覆盖字段。

页面修改：

- 新增/编辑训练页顶部增加“训练计时”区：开始时间、当前耗时、手动修正。
- 详情页显示：训练开始、训练结束、总耗时。
- 动作卡片显示：动作耗时、完成组数。

验收标准：

- 用户不手填时长，也能在保存后看到自动总时长。
- 如果用户手填时长，以手填值为展示主值，并保留自动时长作为辅助信息。

### 4.5 动作练到的部位要更准确

当前 `ExerciseDefinition.category` 只能表达一个大部位，无法表达“主练”和“辅助”。

修改建议：

给 `ExerciseDefinition` 增加：

- `primaryMuscles: List<String>`
- `secondaryMuscles: List<String>`
- `movementPattern: String?`，例如 `push/pull/squat/hinge/carry/core/cardio/mobility`

给 `ExerciseRecord` 保存当时快照：

- `primaryMuscles`
- `secondaryMuscles`

原因：动作库后续可能修改，但历史记录应保留当时选择动作时的定义。

动作库管理页增加：

- 按部位筛选。
- 编辑动作部位。
- 批量检查“未标注精确部位”的动作。

验收标准：

- 统计页可以按精确肌群统计，不只按训练主题统计。
- 历史记录不因动作库后续修改而变形。

### 4.6 增加器械信息

笔记需求：目前只有重量，需要提供器械选项：杠铃、哑铃、弹力带、龙门架、固定器械等。

修改建议：

给 `ExerciseDefinition` 和 `ExerciseRecord` 增加：

- `equipment: String`

推荐枚举：

- `barbell`：杠铃
- `dumbbell`：哑铃
- `kettlebell`：壶铃
- `machine`：固定器械
- `cable`：龙门架/绳索
- `smith`：史密斯
- `band`：弹力带
- `bodyweight`：自重
- `cardio_machine`：有氧器械
- `other`：其他

页面修改：

- 自定义动作弹窗增加器械选择。
- 动作详情小字显示：`背 · 重量 · 龙门架`。
- 编辑训练时允许覆盖器械，因为同名动作可能换器械完成。

验收标准：

- 新增动作必须有器械默认值。
- 自定义动作可选择器械。
- 历史记录保存器械快照。

### 4.7 文件保存位置

笔记需求：文件保存位置。

修改建议：

设置页新增“本地数据”卡片：

- 训练记录文件：`workout_records.json`
- 动作库文件：`exercise_library.json`
- 最近一次备份路径。
- 按钮：`查看保存位置`、`复制路径`、`导出备份`。

技术说明：

- 当前使用 `path_provider` 的 `getApplicationDocumentsDirectory()`。
- Android 上应用私有目录普通文件管理器未必可直接访问，因此“导出备份”和“分享备份”比只展示路径更重要。
- 可考虑新增 `share_plus` 依赖，用系统分享面板把 JSON/CSV 发到网盘或文件管理器。

验收标准：

- 设置页能看到当前数据目录路径。
- 导出成功后能看到备份文件完整路径。
- 用户知道 App 数据是本地离线保存。

### 4.8 日历形式总览

笔记需求：日历形式的总览图。

修改建议：

在 `HistoryPage` 或新增 `CalendarPage` 增加月历视图：

- 有训练的日期显示圆点或色块。
- 色彩区分训练量：轻/中/重。
- 点开日期显示当天训练列表。
- 空日期可直接点 `新增训练`，日期自动带入。
- 月顶部显示：本月训练次数、总容量、总时长。

第一版可以自己实现月历网格，不必立刻引入复杂日历库。

验收标准：

- 月视图能快速看出训练频率。
- 点击有记录日期能进入详情。
- 点击空日期能新增该日期训练。

### 4.9 导入以前的数据

笔记需求：把以前的东西导进去。

修改建议：

先支持两种导入：

1. App 自己导出的 JSON 备份。
2. CSV 模板导入。

CSV 推荐字段：

```csv
date,title,status,bodyParts,exerciseName,category,type,equipment,setIndex,weight,reps,timeSec,distanceKm,speed,side,completed,rpe,note
2026-06-26,背部训练,正常,背,高位下拉,背,weighted,cable,1,45,10,,,,,true,8,
```

导入流程：

- 选择文件。
- 解析并展示预览：总记录数、动作数、异常行。
- 重复检测：优先用 `id`，没有 `id` 时用 `date + title + exerciseName + setIndex` 生成哈希。
- 用户确认后写入。
- 导入失败不覆盖现有数据。

验收标准：

- 可导入当前 App 导出的 JSON。
- 可导入 CSV 模板。
- 导入前有预览，导入后有结果提示。
- 出错时告诉用户第几行、哪个字段有问题。

## 5. 数据模型修改建议

### 5.1 Schema 升级

当前 `StorageService.schemaVersion = 1`。建议升级到 2，并增加迁移逻辑。

原则：

- 旧 JSON 能读。
- 新字段全部有默认值。
- 迁移失败时不删除原始文件。

### 5.2 SetRecord v2

建议字段：

```dart
class SetRecord {
  final int setIndex;
  final double? weight;
  final double? reps;
  final int? timeSec;
  final double? distanceKm;
  final double? speed;
  final String side;
  final bool completed;
  final double? rpe;
  final String? createdAt;
  final String? completedAt;
  final String? updatedAt;
}
```

统计规则：

- `completed == false` 的组默认不计入容量、次数、时长。
- 没有 `completed` 字段的旧数据按 `true` 处理。

### 5.3 ExerciseDefinition v2

建议字段：

```dart
class ExerciseDefinition {
  final String id;
  final String name;
  final String category;
  final String type;
  final String unit;
  final String equipment;
  final List<String> primaryMuscles;
  final List<String> secondaryMuscles;
  final String? movementPattern;
  final bool isUnilateral;
  final bool enabled;
}
```

### 5.4 ExerciseRecord v2

建议字段：

```dart
class ExerciseRecord {
  final String id;
  final String name;
  final String category;
  final String type;
  final String unit;
  final String equipment;
  final List<String> primaryMuscles;
  final List<String> secondaryMuscles;
  final bool isUnilateral;
  final List<SetRecord> sets;
  final String note;
  final String? startedAt;
  final String? finishedAt;
}
```

### 5.5 WorkoutRecord v2

建议字段：

```dart
class WorkoutRecord {
  final String id;
  final String date;
  final String title;
  final List<String> bodyParts;
  final String status;
  final int? durationMin;
  final int? autoDurationSec;
  final String? startedAt;
  final String? finishedAt;
  final String note;
  final List<ExerciseRecord> exercises;
  final String createdAt;
  final String updatedAt;
}
```

## 7. 优先级路线图

### P0：先修正训练记录体验

- 解释并优化完成开关。
- RPE 改为可选高级字段。
- 每组增加 `createdAt/completedAt/updatedAt`。
- 未完成组不计入统计。
- 动作增加器械字段。
- 设置页展示本地保存位置。

完成后，训练中不会再出现“不知道这组做没做”“这个滑块是干嘛的”“RPE 和次数是不是重复”的问题。

### P1：补齐数据准确性和迁移能力

- Schema v2 迁移。
- 动作库增加主练/辅助肌群。
- 动作库管理支持编辑器械和肌群。
- JSON 备份导入。
- CSV 模板导入。
- 日历总览。

完成后，旧数据能进入 App，训练历史能按日期和肌群复盘。

### P2：增强统计复盘

- 动作趋势图。
- 肌群周训练量。
- 器械使用统计。
- 估算 1RM / PR 记录。
- 训练总时长趋势。
- 肌肉恢复热力图的简化版。

完成后，App 不只是记账本，而是能帮助判断训练是否均衡、是否进步。

### P3：训练中辅助能力

- 休息计时器。
- 训练模板/routine。
- 复制上次同动作重量次数。
- 超级组。
- 热身组标记。
- CSV 导出。

这些功能很有用，但建议等 P0-P2 的数据基础稳定后再做。

## 8. 具体实现注意事项

### 8.1 兼容旧数据

所有 `fromJson` 必须容忍字段缺失：

- `equipment` 缺失时按动作类型推断，无法推断则为 `other`。
- `primaryMuscles` 缺失时用 `category` 生成单元素列表。
- `completedAt` 缺失时不显示完成时间。
- `completed` 缺失时按 `true`。

### 8.2 统计口径要统一

建议统一规则：

- 容量 = 已完成重量组的 `weight * reps`。
- 自重次数 = 已完成自重组的 `reps`。
- 时间训练 = 已完成时间组的 `timeSec`。
- 有氧时间 = 已完成有氧组的 `timeSec`。
- 训练时长 = 优先手填 `durationMin`，否则用自动耗时。

### 8.3 本地优先

继续保持本地离线：

- 不加账号。
- 不加远程数据库。
- 导入导出作为迁移和备份方案。

如果未来需要同步，应先把数据模型和迁移做好，再考虑云端。

## 9. 第一轮开发建议拆分

建议第一轮只做 P0，改动可控：
